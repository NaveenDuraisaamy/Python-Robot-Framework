Lesson 1: Robot Framework Basics
What is Robot Framework?
Robot Framework is an open-source test automation framework.

Used for:

Web Testing
API Testing
Database Testing
Desktop Testing
Mobile Testing

It follows a Keyword Driven Approach

Exmples:

Open Browser
Input Text
Click Button
Close Browser

These are called Keywords.
----------------------------------------
Structure of a Robot File

A .robot file contains sections.

*** Settings ***
*** Variables ***
*** Test Cases ***
*** Keywords ***

Think of it like:

Settings     -> Imports
Variables    -> Data
Test Cases   -> Tests
Keywords     -> Functions
------------------------------
1. Settings Section

Used to import libraries.
*** Settings ***
Library    SeleniumLibrary

Similar to Python:
from selenium import webdriver

2. Variables Section

Store reusable values.
*** Variables ***
${URL}    https://google.com
${BROWSER}    chrome

${} means a scalar variable.Usage:
Usage:
Open Browser    ${URL}    ${BROWSER}

3. Test Cases Section

Actual tests.
*** Test Cases ***
Open Google
    Log To Console    Hello Naveen

Output:
Hello Naveen
PASS

4. Keywords Section

Custom reusable functions.
*** Keywords ***
Open Application
Log To Console    App Opened

Usage:
*** Test Cases ***
Test Login
    Open Application

----------------------------------
Indentation Rule

Robot uses 2 or more spaces between columns.

Correct:
Log To Console    Hello
Wrong:
Log To Console Hello

------------------------------------Lesson 2----------------------------------------
Test Cases (Industry Standard)

In real projects, a test case contains:

Documentation
Tags
Setup
Teardown
Assertions
Reusable Steps

1. Test Case Name
*** Test Cases ***
Verify User Can Login

Not:
TC1

Always use business-friendly names.
------------------------------
2. Documentation

Used to explain the purpose of the test.

*** Test Cases ***
Verify User Can Login
    [Documentation]    Verify valid user login functionality
    Log To Console     Login Successful

3. Tags

Used for filtering tests.

*** Test Cases ***
Verify User Can Login
    [Tags]    Smoke    Login
    Log To Console     Login Successful

Run only smoke tests:
1.robot -i Smoke tests
2.robot -i Login tests

4. Setup
Runs before every test.

Example:
*** Test Cases ***
Verify Login
    [Setup]    Log To Console    Opening Browser

    Log To Console    Login Test

    [Teardown]    Log To Console    Closing Browser

Output:
Opening Browser
Login Test
Closing Browser

5. Teardown

Runs after test execution.

Even if the test fails, teardown executes.

Example:
*** Test Cases ***
Demo Test
    Log To Console    Executing Test

    [Teardown]    Log To Console    Cleanup Completed

6. Multiple Tests Example
*** Test Cases ***
Verify Login
    [Documentation]    Login validation
    [Tags]    Smoke

    Log To Console    Login Success

Verify Search
    [Documentation]    Search validation
    [Tags]    Regression

    Log To Console    Search Success

7. Comments

# This is comment

*** Test Cases ***
Demo
    Log To Console    Hello

8. Empty Lines

Use empty lines to improve readability.

Good:

*** Test Cases ***
Verify Login
    Log To Console    Step 1

    Log To Console    Step 2

Most Important Concept
Robot executes:
Step 1
Step 2
Step 3

line by line.
If Step 2 fails:

Step 1  -> PASS
Step 2  -> FAIL
Step 3  -> SKIPPED

unless error handling is used.

---------------Lession 3--------------------
Variables are used to store data.

There are 4 types you must know:
1. Scalar Variables      ${}
2. List Variables        @{}
3. Dictionary Variables  &{}
4. Command Line Variables
--
1. Scalar Variables
Store a single value.

*** Variables ***
${NAME}      Naveen
${CITY}      Chennai
${AGE}       31

Usage:

*** Test Cases ***
Print Details
    Log To Console    ${NAME}
    Log To Console    ${CITY}
    Log To Console    ${AGE}
----------------------------
2. List Variables
Store multiple values.

*** Variables ***
@{FRUITS}    Apple    Mango    Orange

*** Test Cases ***
Print Fruits:
    Log To Console    ${FRUITS}[0]
    Log To Console    ${FRUITS}[1]
    Log To Console    ${FRUITS}[2]

Apple
Mango
Orange

Print Entire List:

*** Test Cases ***
Show List
    Log To Console    @{FRUITS}
Output:
Apple Mango Orange
------------------------
3. Dictionary Variables
Store key-value pairs.

*** Variables ***
&{USER}
...    name=Naveen
...    city=Chennai
...    role=Tester

*** Test Cases ***
Print User
    Log To Console    ${USER.name}
    Log To Console    ${USER.city}
    Log To Console    ${USER.role}
Output:
Naveen
Chennai
Tester
-------------------------
4. Variables Inside Variables

*** Variables ***
${FIRSTNAME}    Naveen
${LASTNAME}     D

${FULLNAME}     ${FIRSTNAME} ${LASTNAME}

Log To Console    ${FULLNAME}
output:
Naveen D
----------------------------
Variable Naming Standards:

${USERNAME}
${PASSWORD}
${BASE_URL}
-----------------------------

------------------------Lesson 6: Locators------------------
Without locators, Selenium cannot find elements.

Think:

Website
  ↓
Locator
  ↓
Element
  ↓
Action

Example:

Google Search Box
        ↓
name=q
        ↓
Input Text

Types of Locators

Most used:

id
name
xpath
css

Interview Priority:
1. id
2. css
3. xpath
4. name

---------
1. ID Locator
<input id="username">
Input Text    id=username    Naveen

2. Name Locator
<input name="q">
Input Text    name=q    Robot Framework

Google search box uses:
name=q

3. XPath Locator
<input name="q">
XPath:
xpath=//input[@name='q']

Usage:
Input Text    xpath=//input[@name='q']    Robot Framework

--------------
XPath Basics

Find by attribute:
xpath=//input[@id='username']
Find button:
xpath=//button[@type='submit']
Find link:
xpath=//a[text()='Login']
--------------
4. CSS Locator

<input id="username">
css=#username
<input name="q">
css=input[name='q']

---------
Which Locator to Use?

Preferred order:

ID
↓
CSS
↓
NAME
↓
XPath

Avoid long XPath whenever possible.

---------
Waits
Wait Until Element Is Visible
Wait Until Page Contains
Wait Until Element Is Enabled
---------

Why Resource Files?

Imagine:
100 test cases
20 common actions

Without Resource Files:
2000+ duplicate lines

With Resource Files:
Reusable keywords
Centralized maintenance

This is how enterprise frameworks are built.
